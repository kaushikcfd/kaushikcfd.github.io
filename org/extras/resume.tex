% {{{ Research

\section*{\Large\textsc{Research}}

\vspace{-4ex}
\rule{\textwidth}{0.1ex}
\begin{description}
  \item[Optimizing Einstein-Summation Subprograms] Einstein-summations provide a
    simple notation to express a wide-range of Linear-Algebra primitives.
    Achieving roofline FLOP-throughput for these operations still remains
    challenging. We employ a combination of techniques from auto-tuning to
    pattern-matching to generate efficient OpenCL code for computational
    kernels containing expressions that have einsum-like memory access
    patterns.

  \item[Array Programming Languages and Intermediate Representations] Array
    Programming Languages have been an important vehicle for driving scientific
    applications from as early as the 1960s. Besides providing a close-to-math
    expressibility, their intermediate representations are closer to SIMD
    architectures making it easier to engineer optimizing compilers targeting such
    hardwares.  \textsc{Pytato} provides one such IR that lowers $n-d$ array
    programs to computation graphs comprising of \textit{pure}-Array Ops that can
    be targeted to OpenCL / CUDA / JAX.

  \item[Near-Roofline Discontinuous Galerkin Action Operators] Discontinuous Galerkin operator
    applications comprise of many fine-grained array operations that can push
    them into the memory-bound regime. With kernel and loop fusion we can bump
    up the workload's Arithmetic Intensity, however, performing fusion might
    also negatively affect the kernel's performance by inhibiting device's
    latency hiding abilities by further introducing dependency edges and
    increasing the working set size of the inner loops. In the
    \textsc{MIRGE-Com} framework we address such trade-offs for GPU systems.

  \item[Finite Element Assembly on GPUs] Evaluation of Finite Element operators
    result in a diverse set of computational kernels making it a difficult problem
    to find one optimization strategy that achieves near-peak performance for all
    the kernels on GPUs.  We solve this problem by using high level code
    generation tools that select the optimization strategy based on the loop
    structure of the kernel.
\end{description}

% {{{ Key Software Contributions

\section*{\Large\textsc{Key Software Contributions}}

\vspace{-4ex}
\rule{\textwidth}{0.1ex}
{\scriptsize P = Primary (Maintainer) | S = Significant contributions}
\vspace{1ex}\\
\begin{tabular}{p{0.18\textwidth} p{0.74\textwidth}}
  \textbf{Loopy}$^S$ &
    An intermediate representation and transformation toolkit for programs with
    single exit-point loops for targets with SPMD execution models. My work
    introduced the callables abstraction within the IR that eventually enabled
    it to interact with the computer algebra systems in projects like Firedrake,
    MIRGE-Com. Documentation: \url{https://documen.tician.de/loopy/}\vspace{1ex}\\

  \textbf{Pytato}$^P$ &
    An intermediate representation for $n$-d array operations that supports the
    pure subset of Numpy operations. The IR can be targeted to Loopy IR,
    XLA/JAX, etc. Pytato has been used to deploy a production Discontinuous
    Galerkin solver which is capable of delivering roofline performance while a
    computational scientist writes Numpy-like code. Documentation:
    \url{https://documen.tician.de/pytato/}\vspace{1ex}\\

  \textbf{feinsum}$^P$ &
    Toolkit for managing databases of code-transformations on Batched Einstein
    Summations. Unlike typical computational primitive libraries that ship
    optimized computational kernel in source or binary form, feinsum records
    code- transformations from a trivial schedule, resulting in relative savings
    from not having to flush the operands to the DRAM. Documentation:
    \url{https://feinsum.readthedocs.io/}\vspace{1ex}\\

  \textbf{Arraycontext}$^P$ &
    Provides a subset of Numpy operations that are avaialable to both lazy and
    eager modes of execution. At runtime, these operations can be retargeted to
    array implementations, such as Pytato, JAX, PyOpenCL, etc.  Documentation:
    \url{https://documen.tician.de/arraycontext/}.\vspace{1ex}\\

  \textbf{symoxide}$^P$ &
  A computer algebra system written purely in Rust. Demonstrated $\sim$5
  speedup w.r.t. state-of-the-art implementations in Python for expression
  traversals of production Scientific Computing workloads.  Documentation:
  \url{https://docs.rs/symoxide}
    \vspace{1ex}\\

  \textbf{ISL-Rs}$^P$ &
  Rust bindings for Integer Set Library to access polyhedral analysis
  primitives, where the implementation is responsible for automatic cleanup of
  objects as per Rust semantics. Documentation: \url{https://docs.rs/isl-rs}.
  \vspace{1ex}\\

  \textbf{pybliss}$^P$ &
  Idiomatic Python bindings for the graph isomorphism toolkit, \textsc{Bliss}.
  Documentation: \url{https://pybliss.readthedocs.io}.
  \vspace{1ex}\\

  \textbf{Miscellaneous} &
    Minor improvements to \textsc{PETSc}, \textsc{PyCUDA}, \textsc{Firedrake}, etc.\\
\end{tabular}
